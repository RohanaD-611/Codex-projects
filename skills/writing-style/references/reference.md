## Reference 1: 找AI算命，它说我还没“进入人生”

快30了，我还在反复问自己，到底可以往哪个具体方向走。

我以为，方向这个东西只要思考得足够缜密，某一刻肯定可以顿悟，或者命中注定，如同动画片里脑袋上的灯泡，一瞬间突然亮起。

我不是一个很有天赋的人，以前那些兴趣爱好也不知道什么时候开始，被当成无用的东西慢慢丢掉了，到现在，我甚至想不起来自己曾经最喜欢做什么。

十年过去，我好像一直在找方向，又一直没有找到；我好像什么都可以做，但又没有一个聚焦点。

迷茫至极，我决定找AI算命。

我把生日、性格、做过的工作、甚至最讨厌的工作场景，一股脑丢给AI，问它：你告诉我，我到底适合什么职业方向。

它回得非常完整。

我第一反应是，哇，好专业。

但看完我心里仍然空空的。它说的都对，但也不痛不痒；它像一个很会说话的人，把所有正确的话都说了一遍。

我换不同AI又问了一遍。

GPT说，你擅长观察、分析和理解，但你容易停在：“理解人生”，而不是“进入人生”。

我突然反应过来，原来我卡住的点是我思考得太多还不敢往前走，而不是不知道方向。

很多人聊个人定位，会从能力盘点开始。

我也是这么做的。

但探索下来，发现真正有用的是另一条线：先看你长期愿意承受什么，然后开始做，先做，再调整。

因为“擅长”会变。

你现在不会的，半年后可能会。

但不愿意承受的东西会让你持续难受，比如我就是受不了高密度无效沟通，或者看到KPI头就开始晕。

后来我跟AI咨询，也不再直接问它要答案，而是让它做我的镜子。

具体我让它梳理了以下几件事：

第一件，让它复盘我过去做过的事，让我回忆那些有能量而不内耗的时刻。

比如同样是写方案，比起执行“漂亮汇报”的内耗，“把复杂问题梳理清楚”的时刻明显让我更有动力。

第二件，让它反向提问我。

我让它刻意问一些不舒服的问题，比如：你有没有对某些人、某种人生状态产生过嫉妒？

我第一次看到这个问题时，沉默了很久。

第三件，它开始纠正我的方向幻想。

AI 跟我说，方法论不是想出来的，要学会“借别人的结构成长”，在模仿中学习，在学习中成长。

它让我从 0 到 1 做一个最小成本的 MVP，从最简单的步骤开始，先把项目流程跑通。

当这个小项目成功时，我从结果中获得了真实的自信。

发布后的数据会告诉我读者真正爱看什么，而我的身体反应则告诉我，自己愿不愿意沿着这个方向继续下去。

现在我对“个人定位”这个话题有了不一样的理解。

比起“找到你是谁”，找到愿意长期重复、且别人确实需要的那一小块价值显得更让人有成就感。

执行前先了解自己，AI在这里的引导作用很大。

但AI并不能替你决定人生，它更像一个效率很高的咨询师，帮你更快排除错误选项，把你从自我想象拉回真实反馈。

总的来说，个人定位没有办法靠“想出来”，需要在日复一日的执行过程“做”出来。

AI给我“算命”时，还说了一句：不要抱有“天命感”，方向盘还是在你手里。

人生并不会因为天赋自动展开，真正决定你会走向哪里的，需要靠具体的行动、积累和调整。

聊完后，方向依然没有十分明确和具体，脑袋上的灯泡依然没有“pia”的一下亮灯，但光好像在越来越清晰。

—— 2026.6.1

## Reference 2: 关于AI时代的职业规划，我看过的最好答案

说到AI时代的职业规划，Up主张咋啦用仙童半导体的故事提出了新的解读角度。当年，从肖克利半导体公司离职的八位人才，创立了仙童半导体。随后，人才从仙童不断外溢，催生了英特尔、AMD、红杉资本、KPCB等改变世界的企业。

这带来一个反直觉的提示：开创一个行业的公司，往往不是该行业长期最成功的公司；真正释放巨大价值的，是那些从老组织中离开、重新组合的人才。由此，产生了一个职业规划的核心转向——从追随公司，转向追随人才。与其挤破头进入当下的明星企业，不如关注那些正在流动的顶尖人物，追踪他们的去向，加入他们即将创造的新东西。

一、另一个被忽略的历史坐标

除了硅谷的仙童传奇，还有一个更贴近我们这一代人的故事：PayPal黑手党。

2002年，PayPal被eBay收购后，它的核心团队成员陆续离开。你很难想象到这群离职员工后来做了什么，埃隆·马斯克创立了特斯拉和SpaceX，里德·霍夫曼创办了LinkedIn，彼得·蒂尔成为了硅谷顶级投资人并参与创立了Palantir，陈士骏和查德·赫尔利创办了YouTube……这份名单几乎可以写满半部互联网创新史。

有意思的是，当年PayPal本身并不是一家“改变了世界”的公司；它只是一个还算成功的支付工具。真正改变世界的，是那群在PayPal共事过然后散落四方的人，这与仙童的故事如出一辙。开创性公司的价值，不只在于它自身能做多大，还在于它成为了什么样的人才枢纽。

二、一个被长期误用的职业逻辑

我们从小被灌输的职业规划是，找一家好公司，然后待下去。

于是大多数人做决策时，眼睛只盯着风口、公司品牌、估值、融资轮次、工牌上的职位。这套逻辑在工业时代可以成立，因为当时的资产是厂房和设备，人只是螺丝钉。但今天，真正的资产在人和人之间的连接，知识和判断力，还有那些敢于离开并重新开始的个体身上。

当你追随公司时，你把命运交给了组织；当你追随人才时，你把主动权留给了自己。

为什么？公司的平均寿命在缩短，而一个优秀人才的半衰期却可以很长。更重要的是，优秀人才会彼此吸引，会形成高密度的信息场。你跟着一个人，可以接触到他背后的网络、判断标准和筛选机会的方式，这也是职业成长中真正值钱的东西。

三、如何追随人才？

重新定义你的关注列表

不只是关注那些头衔闪亮的CEO或网红，还需要重点关注那些正在解决复杂问题的人。他们在哪个团队？他们最近在思考什么？他们离开了一家公司之后去了哪里？这些信息远比公司财报更有价值。

你可以做一个简单的练习，列出你所在领域里你最佩服的五个人，他们不一定是最有名的，但具备较强的判断力，拥有流通的高质量信息，可以追踪他们过去三年的职业轨迹。他们的流动路径，就是行业价值迁移的地图。

把“和谁一起工作”排到决策优先级的第一位

面试时，不要只是再问“公司有什么福利”，改成问直属上级是谁？了解团队里有哪些人的背景让我可以学到东西？如果面试过程中，你感受到对方的信息质量低、逻辑混乱、或者不尊重清晰沟通，直接pass。因为你每天要和这些人待八个小时，他们的认知水平会直接塑造你。

接受短期不稳定以换取“长期网络增值”

追随人才意味着你可能需要频繁地跟着人移动。你加入一家初创公司，它可能六个月后倒闭；你跟着一个老板跳槽，新环境需要重新适应。这看起来不稳定，但如果换一个算法，你每换一次，你的人才网络也在扩展。你带着从上一站积累的判断力和人脉进入下一站，你的复合价值在指数级增长。而那些一辈子待在同一家公司的人，一旦公司出问题，他们的网络往往是孤立的。

最后，自己成为值得被追随的人

追随人才是双向的。只有当你自己也在持续输出清晰的判断、可靠的信息、高质量的工作成果时，优秀的人才才会愿意与你交换价值。所以，不要只做跟随者，要做同行者。在任何协作中，主动推进决策和复盘，你也会慢慢成为别人想要追随的人。

四、回到AI时代

有很多人焦虑，AI会取代什么工作？我应该进大厂还是去创业公司？要不要学编程？

焦虑的背后，其实都默认同一个前提：有一个稳定的“公司-岗位”坐标系。但这个坐标系目前正在逐渐模糊。AI让技能贬值加速，团队的最小有效规模缩小，远程协作常态化。在这样的时代，唯一稀缺的东西是高密度、有信任关系的人才网络。

你不会因为知道要学AI而觉得安全，但你会因为认识五个真正懂AI工程的人、并且他们愿意和你共事而安全。仙童的故事告诉我们，公司的兴衰是常态，而人才的流动是永恒。硅谷是由一次又一次“从旧组织里出走-重组-再出走”的循环定义的，而不是依靠于单家公司。

所以，职业规划更需要的是一份你敬佩的人才名单，而不是一张“未来十年最值得加入的公司”榜单，还有一个让你和他们产生连接的计划，可以给你一种“离开任何组织都能活下去”的信心。

真正的职业生涯，不会写在任何一家公司的工牌上，主要体现在你与哪些人一起解决了什么问题，又一起走向了哪里。

—— 2026.6.3

## Latest Revision Reference 1: 5分钟从0到1打造第一个Skill的整个过程

This article is the user's revised version of an AI-use diary. It is the strongest reference for 3.0 rules about concision, avoiding repetition, and keeping the main process article focused.

当我有一个想法需要思考能不能落地的时候，会习惯性去找各种各样的平台聊天、咨询，比如 AI、小红书、知乎，然后自己将所有的信息整合在一起，分析解决方案和步骤是什么；但有时候我甚至搞不清楚自己想要什么，一通乱查后再进行聚焦，等收集到很多信息以后才发现，这个方向可能一开始就不太适合。

这个过程很耗时间，也有可能把我带偏。因为信息多了，表面上好像选择变多了，但真正要做决定的时候，反而更难判断第一步应该从哪里开始。尤其是当一个想法还只是一个没有明确线索的头绪时，如果直接跟 AI 聊，只要你给一点方向，聪明又勤快的AI 同志通常会很快帮你铺开十条路，并告诉你功能、平台、路径和执行步骤分别如何，这种带有迷惑性的完整通常让我误以为自己已经想清楚了，但我知道其实还有一个最关键的问题需要明确：我到底想解决什么问题。

当我有一个模糊想法的时候，我希望它可以引导我澄清问题，并告诉我落地框架，这对我太重要了。我需要让它先慢下来，一步步确定我的本质问题。追求本质是我的mantra，很多事情如果本质问题没找准，后面做得越多，最后可能只是精致地跑偏。

今天我突然想到，为什么不用一个想法 skill 辅助我思考呢？

所以我一开始给 Codex 的 skill 要求也很明确，不需要直接给我一个完整的方案，也不需要先推荐很多平台。你现在的任务不是给方案，而是像一个 idea coach 需求顾问一样，辅助我做需求判断和决策分析。每轮的问题优先引导我思考最终要解决什么问题，哪一part最需要先做，以及结果需要输出什么。

后来我根据自己的需求又想到一个问题，idea coach skill 要不要拆成三个独立 skills？严格需求顾问、产品经理教练和Skill 架构师，各司其职，然后再由一个 idea coach agent 把它们串起来。我认为这样可以实现：先有人帮我梳理清楚需求，再有人帮我判断 MVP，最后有人帮我把工作流变成真正可复用的 skill。

但Codex建议我，你的1.0 版本不一定要做得这么复杂。一个还没有跑通过的工具，一上来就拆成很多模块，后面维护起来可能会很麻烦。最后我们决定先做成一个整合skill，里面包含需求顾问、产品经理教练以及Skill 架构师这三种模式，等以后真的用多了再判断要不要拆分。这样至少第一版可以先跑通，而不是一开始就陷入架构幻想（INTJ真的很容易这样）。我们明确了这三个模式分别的职责范围：

严格需求顾问解决的是想法还很模糊的时候，它可以先帮我确认核心痛点、最小范围、输入数据、处理流程、输出位置等等信息。它更像是在跟我说，这个想法值不值得落地？现在最需要先做哪一个功能？我需要给 AI 什么材料去支持它？

产品经理教练解决的是另一个问题：当一个方向已经大概成立，但想实现的功能太多的时候，它可以帮我判断优先级和 MVP。哪些是第一版必须有的功能，哪些是不太必要且比较难维护的。这个环节对我也很重要，因为我经常会在脑子里把一个小东西想成一个完整系统，思考起来很兴奋，做的时候就开始发现落不了地。

Skill 架构师则是最后一层翻译。一个想法哪怕已经被梳理成工作流，也不等于它可以直接变成 Codex skill。它还需要被明确为具体触发条件、输入要求、追问流程、确认节点、输出模板及文件结构等等，甚至要判断哪些内容应该写进 SKILL.md，哪些应该放到 references/。这一步有点像把一个人的思考习惯，翻译成另一个 AI 也可以读懂并稳定执行的说明书。

根据我的设定，它的默认流程也不是一路自动跑到底，而是每个阶段结束后都会停下来等我确认。比如先输出需求澄清文档，然后问我是否进入产品经理教练模式，帮我判断优先级并收束最小范围；等我确认后，再继续输出 MVP 工作流文档；之后再问我要不要进入 Skill 架构师模式，把工作流转成 skill 结构和 SKILL.md 草稿。

这个停顿对我来说也是它有价值的地方。

因为我不想要一个自动狂奔的 agent。我想要的是一个能陪我思考，但每个关键节点都让我自己进行判断的工具。它可以帮我问问题、整理信息和形成文档，但最后要不要继续往下，还是由我自己来决定。这样它才像一个辅助思考的工具，而不是又一个把我推着往前走的系统。

如果要说这个 skill 和市面上常见的 idea coach 有什么区别，我觉得它更像一个收束工具，它通过反复把我拉回几个朴素的问题，问我到底要解决什么？希望它扮演具体哪个角色？告诉我需要给它提供什么材料，让我决定中间可以怎么处理等等；问题直白到曾经让我有点烦，但也让我对落地有了真正的思考。当一个INTJ想法很多的时候，最缺的也往往是这样的落地结构。没有结构的时候，AI 给得越多，我会越迷茫；有了结构以后，就算第一版很小，起码也感受到自己可以跑通了。

这个skill做完后，我对 AI 使用也有一点新的感觉。以前我觉得用好 AI 的关键是写一个好 prompt，后来发现 prompt 只能解决一次问题，而 skill 更像是把我的某种思考方式固定下来，让它下次还能继续帮我工作。

Idea coach 是我用这个思维生成的第一个skill，对我来说，它就是这样一个起点。

## Latest Revision Reference 2: Skill 落地过程的一些小问题

This article is the user's revised follow-up piece. It is the strongest reference for splitting technical landing issues into a separate shorter article.

我的第一个skill —— Idea coach，真正落地的时候其实出现了一些很现实的小问题。比如 skill 文件里面如果直接写太多中文，在 Windows 下跑校验脚本时会因为编码问题报错，所以最后 SKILL.md 主体用了英文指令，但要求输出时按用户语言来。这个细节还挺有意思的，它让我意识到做一个 skill 和写一段 prompt 不一样，prompt 只要当前能用就行，但 skill 要考虑后续能不能被验证、能不能被复用、能不能在新的对话里继续被调用。

然后我听某些博主的劝告，又把skill存到了 GitHub，后续版本迭代在GitHub里面进行，防止改烂了回不到上一个版本。这个过程对我这种 GitHub 小白来说，也算是顺手补了一课。我原来不太清楚仓库 URL、HTTPS token、SSH key、GitHub CLI 这些概念到底有什么区别，也不知道 Codex 读取 GitHub 的逻辑是什么。后来才理清楚，Codex 并不能够直接读取我的 GitHub 账号，它操作的是我电脑上的本地 Git 仓库；要上传到 GitHub，需要先本地 commit，再绑定远程仓库，然后 push 上去。

这些过程本身不算是 idea coach 的核心功能，但它让我感觉这个 skill 不再只是某次聊天里生成的一段文字，而是一个可以被保存、更新和迭代的小工具。后来为了让新对话也能直接调用 Idea Coach，我又把它安装到 Codex 全局 skills 目录里。再后来，为了避免每次更新都复制一遍，还改成了目录链接，让全局目录直接指向 GitHub 仓库里的那一份源文件。

这样以后我只需要维护仓库里的 skills/idea-coach/，Codex 全局也会读取同一份。它不再是一次性生成完就结束的产物，而是真的变成了我工作流里可以长期使用的一小块东西。对我来说，这种跨越历史的感觉还挺神奇。
